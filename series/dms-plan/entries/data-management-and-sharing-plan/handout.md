# Data Management and Sharing Plan

**From project systems to repository delivery**

## Core message

A Data Management and Sharing Plan is a living workflow, not a compliance attachment that sits quietly in a proposal file.

Its real purpose is to describe what will actually happen to the data over time:

- how it will be collected
- how it will be organized
- how it will be documented
- how it will be transformed
- how it will be validated
- how it will be shared

That is where Steve Slevinski adds practical value.

He helps researchers and project teams move from local project systems toward a dataset that can be:

- understood
- cleaned
- structured
- documented
- validated
- delivered

A DMS Plan is not just a writing exercise. It begins a practical operational path.

## Why this matters now

Many teams already know they need a DMS Plan.

Many also know where they are ultimately headed, whether that is NDA or another repository.

The hard part is everything in between.

That middle work often includes:

- gathering files from multiple systems
- deciding what counts as source data
- clarifying which files are primary and which are derivative
- cleaning and normalizing structure
- building dictionaries and documentation
- harmonizing variables
- validating delivery packages
- preparing repository-facing outputs
- repeating the process when updates are required

This is the messy middle of research data work.

The DMS Plan only becomes meaningful when this middle layer is understood and supported.

## The DMS Plan as a living guide

Every research project should develop a DMS Plan that evolves over time and remains aligned with NIH policy.

The practical question is not just whether a plan exists.

The practical question is whether the plan remains useful as:

- the project grows
- the data changes
- the systems change
- the repository requirements become more concrete
- deadlines begin to matter

The DMS Plan should stay connected to:

- the real project workflow
- the actual data sources in use
- the tools the team depends on
- the standards that matter for the project
- the destination repository and its specific demands

That is how the guide stays real instead of merely aspirational.

## The real DMS environment

The DMS environment is usually broader than people initially describe.

It may include:

- the researcher and principal investigator
- the project team and coordinators
- project systems such as REDCap
- spreadsheets and CSV exports
- external tools such as NIH Toolbox
- imaging or technical data sources
- outside entities that hold or generate data
- the repository that will receive the final deliverable form of the dataset

This matters because the work is not about one person cleaning one folder.

It is about understanding how these pieces connect, where the hidden structure problems are, and what has to be built between collection and sharing.

This work helps the project team understand and shape that path.

## The actual data lifecycle

The clearest way to understand the work is through the lifecycle:

- source data
- raw structured data
- derivatives
- repository delivery

### Source data

Source data is the original material collected across the project.

It may include:

- REDCap exports
- NIH Toolbox files
- spreadsheets
- CSV exports
- technical system outputs
- files from collaborators or vendors

Source data is often broad and heterogeneous. It may contain:

- more fields than are needed later
- inconsistent value conventions
- unclear naming
- overlapping versions
- local assumptions that are not documented

That means source data is valuable, but it is rarely ready for analysis or sharing in its original form.

### Raw structured data

Raw structured data is the cleaned and organized form prepared for analysis and later sharing.

This stage usually requires:

- consistent naming
- normalization of values
- standardized table structure
- clearer variable meaning
- stronger documentation
- explicit data dictionaries

This is often one of the most labor-intensive parts of the workflow.

Projects frequently underestimate how much work it takes to move from:

> “we have the files”

to:

> “we have structured data that another person can understand and that a repository can accept”

### Derivatives

Derivatives are outputs generated from the raw structured data.

They may include:

- reports
- visualizations
- models
- harmonized tables
- submission-ready repository files

Some derivatives exist for analysis.

Others exist because external sharing requires them.

Projects often need both, and the distinction matters because teams sometimes build analysis-facing outputs without fully preparing repository-facing outputs.

### Repository delivery

Repository delivery is where planning becomes concrete.

For repositories such as NDA, this may include:

- harmonizing variables to repository expectations
- validating files and dictionaries
- preparing submission packages
- resolving errors discovered during validation
- managing recurring submissions and updates

Repository delivery is not a final small step.

It often requires additional structure, additional documentation, and additional technical review late in the process.

## Standards and structures

Standards matter because they make data more understandable, reusable, and transferable.

Depending on the project, relevant standards may include:

- BIDS where appropriate
- CSV plus clear data dictionary structure
- repository-specific variable alignment
- repeatable naming conventions
- consistent file layout and organization rules

The aim is not to impose formality for its own sake.

The goal is to make the data:

- understandable to others
- easier to validate
- easier to revisit later
- easier to finish successfully

This work can help teams decide which structures matter and how to use them coherently.

## Tools and workflows

The DMS Plan becomes real through tools and workflows.

Typical tools may include:

- REDCap
- Excel
- CSV workflows
- Bash
- Python
- documentation and data dictionaries

Each tool has a role.

For example:

- REDCap may help with data capture and reporting
- Excel may help with review, quick restructuring, and macros
- CSV workflows may help with portability and repository preparation
- Bash and Python may become necessary when the work is too repetitive or too complex for manual handling

The goal is to shape and support these workflows so they become:

- clearer
- more repeatable
- more aligned with sharing goals
- less fragile under deadline pressure

## Repository delivery as real labor

Repository delivery is where many plans become real or fall apart.

For repositories such as NDA, the work often includes:

- matching local variables to the repository dictionary
- clarifying required versus optional structure
- preparing files in the right shape
- correcting validation failures
- coordinating recurring submission cycles

The DMS Plan is operational, not merely descriptive.

A good DMS Plan should anticipate:

- what local systems produce
- what transformations are required
- what repository constraints are likely to matter
- where labor and responsibility will fall

The point is to do more than discuss those questions and actually carry the work through.

## Timing and project pressure

DMS work is not evenly distributed across a project.

Pressure often spikes around:

- biannual repository cycles
- annual review cycles
- urgent analysis requests
- sudden sharing needs

A team may appear to be doing fine during quiet periods and then struggle when a real submission or deliverable is due.

That is one reason this work matters so much.

This kind of support can be valuable both in:

- planning ahead while the project still has time
- helping the project respond effectively when pressure rises
