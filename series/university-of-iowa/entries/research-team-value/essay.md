# Research Team Value

**Practical DMS Plan, workflow implementation, and repository delivery value for the University of Iowa**

## Core message

Steve Slevinski is practical research capacity for active projects.

His value is present-tense and useful now:

- Data Management and Sharing Plan development and refinement
- workflow implementation across complex project systems
- technical support in code, structure, and validation
- help moving real project data toward real repository delivery

The first case is simpler:

> Steve helps research teams move from fragmented systems and good intentions to structured workflow execution and successful delivery.

The focus here stays on that practical case.

This case is grounded in real support around serious NIH-funded research, including projects with complex data, compliance expectations, repository pressure, and multi-system workflows. The value is not hypothetical consulting language. It comes from work where the data has to be finished, validated, documented, and delivered.

## The problem Steve helps solve

Many projects can state their scientific aims clearly.

Many can also state their repository target clearly.

The difficult part is the work in between.

That work often includes:

- multiple source systems
- inconsistent exports
- unclear data ownership boundaries
- limited documentation
- evolving variables and structures
- repository rules that arrive later than teams expect
- recurring deadlines that create concentrated pressure

This is the workflow implementation layer of research data work.

It is where planning, technical structure, documentation, and repository readiness all collide.

Projects often struggle here because:

- no single person owns the full path
- the workflow is spread across too many systems
- the technical work is too detailed for ad hoc handling
- repository delivery is underestimated until late in the project

Steve is valuable because he can work in that operational gap.

## What this looks like in real projects

Research data rarely lives in a single clean system.

A project may involve:

- REDCap reports and exports
- spreadsheets maintained by coordinators
- CSV files produced through downstream processing
- NIH Toolbox outputs
- imaging or scanner outputs
- files from vendors or external collaborators
- repository-specific dictionaries and validation requirements

All of this has to become:

- understandable
- organized
- documented
- validated
- deliverable

That means the work is not just scientific. It is infrastructural and operational.

Steve's value sits directly in that operational layer.

## Data Management and Sharing Plan value

A DMS Plan is only useful if it stays connected to what the project will actually do.

That means the DMS Plan is not just a grant attachment.

It is a working guide for:

- how data is collected
- where it lives
- what transformations it requires
- what standards matter
- what documentation is needed
- what the repository will require
- what the project team has to do over time to stay aligned

Steve can help with:

- drafting or refining DMS language
- turning policy requirements into practical workflow decisions
- helping teams identify what data really exists and in what shape
- updating the plan as the project evolves
- connecting the plan to actual delivery requirements rather than vague intentions

This matters because a project can have a formally acceptable DMS Plan and still be operationally unprepared.

Steve helps reduce that gap.

## Workflow and code value

Steve's value is also technical and workflow-oriented.

That may include:

- designing or clarifying REDCap report workflows
- organizing CSV exports and table structure
- building or refining data dictionary logic
- using Bash or Python for repeated transformations
- helping separate source data, raw structured data, and derivative outputs
- supporting naming, repeatability, and file organization

This kind of work is often invisible until it is missing.

A project may appear to be functioning while:

- files are hard to reconcile
- variables are not harmonized
- documentation is incomplete
- outputs are not repeatable
- no one is fully sure what the repository will accept

Steve's usefulness is that he can help convert those unclear situations into more structured and repeatable ones.

## Repository delivery value

Repository submission is where many projects finally discover how much unfinished structure remains.

For repositories such as NDA, this may require:

- variable harmonization
- repository-aligned data dictionary structure
- validation against expected rules
- issue resolution during submission
- recurring submission cycles
- support for updates over time rather than one final push

This means repository work is not a trivial final stage.

It is often where:

- planning quality is tested
- structure quality is exposed
- missing documentation becomes a problem
- hidden technical labor becomes unavoidable

Steve can help projects through that process.

His value includes execution as well as planning.

## Why this can matter across multiple projects

This is not a niche one-project contribution.

The same kind of capacity can help multiple research teams because the underlying institutional pain points are often similar:

- DMS expectations are increasing
- repository delivery is more serious than many teams initially assume
- mixed systems create repeated workflow problems
- documentation and structure are rarely finished automatically

This kind of work is best understood as a reusable cross-project function:

- DMS Plan support
- workflow implementation support
- repository-facing technical support
- structure and validation support

That makes the role shared capacity rather than one more isolated project favor.
